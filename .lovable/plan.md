## Two fixes

### 1. Rolling summary instead of full chat history (anti-hallucination + cost)

You're right — for a legal app, hallucination risk is the priority, and a long raw transcript actually *increases* drift on cheap/fast models. The fix is a **rolling conversation summary** that is kept faithful to facts, with only the most recent turns sent verbatim.

**How it will work**

```text
[ system prompt: Bhramar persona ]
[ case file: complaint, notes, document text, payments ]   ← unchanged, ground truth
[ CONVERSATION SUMMARY SO FAR:
   - Client is X, located in Y
   - Bhramar advised filing under Section Z
   - Outstanding question: did client receive notice on date D?
   ... ]
[ last 6 messages, verbatim ]
[ new user message ]
```

- The summary is **factual only** — no opinions, no invented citations. The summarizer prompt will explicitly say: *"Do not add any legal conclusion that was not already stated. If unsure, omit."*
- Anything older than the last 6 turns gets folded into the summary; the recent 6 stay verbatim so tone and immediate context are preserved.
- The case file (complaint, notes, document text, payments) is still sent in full every time. That is the ground truth and is the strongest hallucination guard we have.

**Where it lives**

- `conversations` table → add column `summary text` (nullable). Migration.
- New edge function `summarize-conversation` — runs on Groq llama-3.3-70b (cheapest reliable option) every time the message count past the recent-window grows by 4. Returns a tight bullet list, replaces `conversations.summary`.
- `supabase/functions/chat/index.ts`:
  - Load `conversations.summary` for the active conversation.
  - Inject it as a `[CONVERSATION SUMMARY]` block right after the case file.
  - Frontend keeps sending full history (no client change needed), but the edge function trims it server-side to the last 6 turns before forwarding to the model.
  - Trigger `summarize-conversation` asynchronously (fire-and-forget) when older messages exist.

**Why this reduces hallucination**

- Less raw text to "interpret" on each call → model sticks closer to facts.
- Summary is regenerated by a single deterministic prompt that forbids invention, so noise from earlier model riffs gets filtered out instead of accumulating.
- Case file stays canonical; summary is a *secondary* memory, not the source of truth.

**Cost impact**

- Per-turn input tokens drop roughly 60–80% on long conversations.
- Summary regeneration runs on Groq (~$0.59/M tokens) maybe once every 4 turns — negligible.

### 2. Fix chat not scrolling

Root cause: the center column uses `flex-1 flex flex-col` but is missing `min-h-0`. In a flex container, children need `min-h-0` for inner `overflow-y-auto` to actually scroll — otherwise the child expands to its content and the scroll area never activates. This is the classic flexbox overflow trap and matches what you're seeing on `/app`.

Change in `src/pages/Dashboard.tsx`:
- `<main className="flex-1 flex flex-col min-w-0">` → add `min-h-0`
- `ChatBody` root `<div className="relative flex-1 overflow-y-auto aurora-bg">` → add `min-h-0`

That's it for the scroll fix — purely CSS.
